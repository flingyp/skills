# 爆款文案风格指南

## 核心原则

### 1. 开篇钩子（前3秒）
- **直击痛点**：直接说出用户最痛的问题
  - "这个工具让你在上下班路上都能编程"
  - "写完的代码不会部署"
- **制造紧迫感**：暗示如果不用你就落后了
  - "再不试试这个框架你就落后了"
  - "后端开发圈正在悄悄换赛道"
- **夸张承诺**：给出超越预期的效果
  - "解放双手"
  - "直接上线"
  - "几百行→几十行"

### 2. 内容结构（黄金公式）

**A版公式（痛点型）：**
痛点描述 + 对比传统 + 功能亮点 + 流行语收尾

**B版公式（焦虑型）：**
制造焦虑 + 工具名称 + 功能密集 + 反差对比 + 情感共鸣

**C版公式（目标型）：**
痛点 + 目标宣言 + 功能罗列 + 形容词加持 + 数据背书

### 3. 流行语和形容词库

#### 高频流行语（标注适用年份和场景）

**2024-2025年常用：**
- 纯牛马（2024+适用，适合开源/开发语境）
- 真香（2024-2025年，通用褒义词）
- 战神（2024-2025年，适合性能/效率类工具）
- 福音（2024-2025年，适合入门级工具）
- 爽（2024+，持续通用）

**2025-2026年新增/流行中：**
- 绝绝子（2025年，强烈褒义）
- 炸裂（2025年，强调震撼效果）
- 太顶了（2025年，表示非常厉害）
- 拿捏了（2025年，表示完全掌控）
- 真的会谢（2025年，表示感谢/认可）

**稳定通用型：**
- 高手里的战斗机（2023-2025年，稳定）
- 开源自战神（2023-2025年，稳定）
- 让人头大（2024+，持续通用）
- 一键搞定（2024+，持续通用）
- 直接拿来用（2024+，持续通用）

**流行语使用场景映射：**
- 配置管理类工具 → 纯牛马、真香、爽
- 性能类工具 → 战神、炸裂、太顶了
- 入门级工具 → 福音、绝绝子
- 开源项目、开发者工具 → 纯牛马、高手里的战斗机
- 通用工具 → 真香、拿捏了、真的会谢

#### 夸张形容词库
- 高端AI、真纯、巧妙
- 直接拿来用、轻松上线
- 一大把功能、几乎都能
- 太顶了、炸裂、绝绝子

#### 情感词汇库
- 解放双手、一样的爽感
- 悄悄换赛道、落后了
- 巧妙的避开、让人头大
- 拿捏了、真的会谢

#### 流行语组合禁忌
- ❌ 一句话内堆砌超过3个流行语
- ❌ 使用明显过时的梗（如"我太难了" - 2019年）
- ❌ 混合不同年龄段的流行语（如"666"+"绝绝子"）
- ✅ 推荐1-2个流行语自然融入
- ✅ 选择与工具调性匹配的流行语

#### 流行语新鲜度维护
- **每季度更新一次**流行语库
- 删除明显过期的词汇（超过1年不常用）
- 添加新兴网络用语（TikTok/抖音热榜）
- 标注适用场景和年份

#### 技术术语口语化翻译表

**配置管理类：**
- MCP服务器 → 插件管理/扩展
- SQLite+JSON双层数据库 → 双重保障配置存储
- 配置文件同步 → 多设备配置互通
- 原子写入配置 → 配置不丢不损坏

**开发工具类：**
- 函数调用 → 直接调用
- 云端IDE/环境 → 在线编程环境
- SDK集成 → 一键接入
- 热重载 → 修改即时生效

**部署/运维类：**
- 容器化部署 → 一键打包上线
- 负载均衡 → 自动分流
- 链路追踪 → 出问题快速定位
- 滚动更新 → 更新不中断

**框架/类库类：**
- 声明式路由 → 写路由超简单
- 中间件 → 插件式功能
- 响应式编程 → 自动更新
- 依赖注入 → 自动管理依赖

**性能优化类：**
- 增量编译 → 只改哪里编译哪里
- 懒加载 → 用到再加载
- 虚拟滚动 → 大列表不卡
- 缓存策略 → 访问超快

**通用原则：**
- 用户视角 > 技术准确：说"一键切换"不说"自动重写JSON"
- 场景化表达 > 技术术语：说"多设备同步"不说"云端配置存储"
- 效果导向 > 实现细节：说"快"不说"延迟降低50%"
- 避免技术细节：不说"基于SQLite+JSON架构"，说"配置安全可靠"

**口语化原则：**
1. **去技术名词**：MCP → 插件、SDK → 一键接入
2. **场景化**：原子写入 → 配置不丢不损坏
3. **效果化**：增量编译 → 只改哪里编译哪里
4. **简化表达**：响应式编程 → 自动更新
5. **用用户熟悉的词**：不用"声明式"，用"超简单"

### 4. 对比技巧

**传统 vs 新工具对比：**
- "相比传统的AI编程命令型工具，它不仅..."
- "让部署变得像写代码一样爽"
- "以前用typescript...现在用它几十行就搞定"

**数量对比：**
- PC端+移动端
- 几百行→几十行
- 6万多星、上万星、几千星

**效果对比：**
- 不会部署→直接上线
- 不能随时随地→上下班路上都能

### 5. 功能表达策略

**罗列式（适合功能多）：**
用顿号分隔，快速展示多个功能："数据库管理、实时监控、用户权限控制、链路追踪、备份"

**场景式（适合核心功能）：**
结合使用场景："这意味着你可以同时在PC端还有移动端一起干活"

**对比式（适合优势突出）：**
"几行代码就能搞定...巧妙的避开了rust语言的复杂"

### 6. 数据使用原则

**GitHub星标作为核心背书：**
- "开源6万多星"（5万+）
- "上万星"（1-5万）
- "几千星"（1000-1万）

**数据位置：**
- 放在文案最后作为收尾
- 或放在关键功能介绍后

**数据表达方式：**
- 使用"颗星"、"多颗星"
- 避免精确数字，用"上万"、"6万多"

### 7. 句式节奏控制

**节奏感：**
- 第一句：短促有力（15-20字）
- 第二句：对比展开（30-40字）
- 第三句：功能密集（40-50字）
- 第四句：高潮收尾（20-30字）

**口语化特征：**
- 多用"了"、"了"、"了"（结尾语气助词）
- 少用华丽辞藻，多用大白话
- 使用网络用语和梗

**否定式表达：**
- "不会部署" → "直接上线"
- "头大" → "几十行搞定"
- "复杂" → "一样的爽感"

### 8. 禁忌事项

**❌ 避免学术化表达：**
- 不要长句长段落
- 不要专业术语堆砌

**❌ 避免过度夸张失真：**
- 功能描述要真实
- 数据要准确

**❌ 避免平淡无亮点：**
- 必须有至少1-2个流行语
- 必须有数据背书

## 独创风格定位

### 风格特色：科技感+网络梗+情感共鸣

**与现有风格差异：**
- 更强调"爽感"和"解放"
- 加入"悄悄"、"巧妙的避开"等心理暗示
- 更强烈的对比反差（数量级对比）

**适用场景：**
- 开源工具/项目推广
- 技术框架介绍
- 开发者工具推荐

**目标受众：**
- 程序员、开发者
- 技术爱好者
- 开源项目关注者
